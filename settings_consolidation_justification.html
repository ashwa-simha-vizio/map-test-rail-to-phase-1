Consolidation Techniques

Five engineering techniques enable the 6.9:1 consolidation:

1. Duplicate Elimination
Manual suites repeat identical TCs across model variants or test configurations. Example: "Launch Network Settings from troubleshooting" appears 8 times (different troubleshooting entry points) → TC-SYS-013 parameterizes the entry point.
Impact: 8→1, 4→1 (network rescan repeats), 6→1 (OOBE launch repeats)
2. Parameterized Input Matrix
Tests that vary only by input parameter (band, encryption, button type) become a single TC with a parameter loop. Example: WiFi 2.4/5/6 GHz × WPA2/WPA3/SAE → TC-SYS-011.
Impact: 9→1 (WiFi bands), 8→1 (dismissal buttons), 5→1 (languages)
3. Menu Tree Traversal
Multiple TCs each verifying one menu item's existence are consolidated into a single traversal that asserts all items. Example: HDMI sub-settings → TC-SYS-017.
Impact: 6→1 (HDMI items), 5→1 (Antenna options), 10→1 (DHCP/Security Type)
4. Workflow Consolidation
Sequential related steps (toggle ON → verify → toggle OFF → verify) split into separate manual TCs are combined into one end-to-end flow.
Impact: 10→1 (CC toggle + modes + default), 8→1 (TalkBack + Zoom lifecycle)
5. Context Parameterization
Same assertion tested in different system states (online/offline, OOBE stages) becomes one TC that iterates through states.
Impact: 6→1 (TV Info at OOBE stages), 10→1 (Quick Settings states/deep-links)

Risk Assessment

ConcernResponse
Is coverage being reduced?No. Every single 3P TC ID maps to exactly one automation TC. The 30 TCs collectively test all 206 behaviors.
How can 30 TCs cover 206 behaviors?Manual TCs are written per-instance (e.g., "test 2.4GHz", "test 5GHz", "test 6GHz" = 3 TCs). Automation parameterizes: one TC loops through [2.4, 5, 6] GHz. Same coverage, fewer TC definitions.
What about repeated TCs in manual suite?The manual suite has significant repetition: 8× "Launch Network Settings from troubleshooting", 4× "SSID rescan behavior", 6× "Launch OOBE from OSD". These test identical flows from different models/contexts. Automation runs once, covers all.
Will failure reporting be granular enough?Yes. Each parameterized iteration reports independently. If "5GHz WPA3" fails but "2.4GHz WPA2" passes, the report shows exactly which parameter failed — same granularity as separate manual TCs.
What if we need to add new scenarios?Add new parameters to existing TCs (e.g., new encryption type → add to TC-SYS-011 matrix) or create TC-SYS-031+ for genuinely new functional areas.

Conclusion

30 automation TCs = 206 manual TC behaviors. Zero coverage loss. 100% traceability. 22.8× faster execution. Suitable for nightly regression.
